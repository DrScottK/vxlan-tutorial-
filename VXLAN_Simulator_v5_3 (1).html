Cafe sends ICMP Echo Request to Beef (192.168.11.12). Frame travels on VLAN 10 access port to VTEP-1. VTEP-1 looks up Dst MAC in BGP EVPN table — Beef reachable via VTEP-2 (10.2.2.2), L2VNI 10000.

Step 4: VTEP-1 — VXLAN Encapsulation (L2VNI)
VTEP-1 encapsulates with L2VNI 10000: Outer IP Src=10.1.1.1, Dst=10.2.2.2, UDP 4789, VNI=10000. Original Ethernet frame including inner MACs is preserved intact. 98→148 bytes.

Step 5: VTEP-1 → CORE-1 → VTEP-2
VXLAN packet travels leaf→spine→leaf. VTEP-1 up to CORE-1, CORE-1 routes to VTEP-2. Spine sees only outer IP headers — L2VNI 10000 is opaque. TTL 64→63→62.
VTEP-1→CORE-1: TTL 64→63 | CORE-1→VTEP-2: TTL 63→62 | VNI 10000 opaque to spine

Step 6: VTEP-2 Decap → Beef
VTEP-2 receives VXLAN packet, matches outer Dst 10.2.2.2, UDP 4789 triggers NVE. Strips 4 outer headers. VNI 10000 → VLAN 10. Delivers original frame to Beef on Eth1/1. 148→98 bytes.
VNI 10000→VLAN 10 | 148→98B | Beef receives ICMP Echo Request

Step 7: Beef → VTEP-2 → CORE-2 → VTEP-1 → Cafe
Beef sends ICMP Reply. VTEP-2 encapsulates with L2VNI 10000 (Src=10.2.2.2, Dst=10.1.1.1). Return path uses CORE-2 (ECMP). VTEP-1 decapsulates, delivers reply to Cafe. Round-trip complete.
VTEP-2→CORE-2→VTEP-1 | VNI 10000 | TTL 64→63→62 | ECMP via CORE-2

What Is BGP EVPN?
EVPN — Ethernet VPN (RFC 7432) is a BGP address family (AFI 25 / SAFI 70) that carries Layer 2 and Layer 3 reachability information across a network fabric. Before EVPN, VXLAN used "flood and learn" — unknown traffic was flooded everywhere until the destination was found from the data plane. This does not scale.

The problem it solves: Instead of flooding to find a MAC address, each VTEP advertises its locally learned MACs and IPs as BGP routes. Remote VTEPs install these routes and forward directly to the correct VTEP — no flooding, no unknown unicast storms, no ARP broadcasts crossing the fabric.

How it works: EVPN extends MP-BGP with new NLRI types called EVPN Route Types. Each type carries different information — MAC addresses (RT-2), VTEP presence (RT-3), subnet prefixes (RT-5). BGP extended communities carry Route Targets that control which VTEPs import which routes, providing multi-tenant isolation.

Why BGP? BGP already solves the scale problem — it handles millions of prefixes, supports rich policy via Route Targets, has built-in loop prevention, and uses Route Reflectors to eliminate full-mesh peering. The same protocol that routes the global internet now distributes MAC addresses inside a data center.

EVPN = control plane  |  VXLAN = data plane. EVPN tells every VTEP where every host is. VXLAN carries the actual traffic. Together: a scalable, standards-based, multi-tenant data center fabric.
Route Types Used
RT-2 MAC/IP Advertisement
Advertises a host MAC address, its IP address, and the L2VNI it lives in. Used by VTEPs to build their MAC/IP tables. Enables ARP suppression — remote VTEPs answer ARP locally without flooding.
RD | MAC | IP | L2VNI | VTEP-loopback | Route-Target
RT-3 Inclusive Multicast Ethernet Tag (IMET)
Advertises VTEP presence in a VNI. Builds the ingress replication list for BUM traffic. Replaces PIM multicast entirely.
RD | VTEP-loopback | VNI | Route-Target
RT-5RFC 9136 IP Prefix Route
Defined in RFC 9136 (IP Prefix Advertisement in EVPN). Advertises an entire IP subnet reachable via a VTEP (IP Prefix Advertisement in EVPN). Used with Symmetric IRB to advertise VRF routes across the fabric. Remote VTEPs install these in their VRF routing table and use the L3VNI to reach them. This route type is what enables inter-VLAN routing without each VTEP needing to know every individual host MAC — only the subnet prefix via the L3VNI.
RD | IP-prefix/len | GW-IP | L3VNI | VTEP-loopback | Route-Target
L2VNI vs L3VNI — Key Difference
L2VNI
Carries original Ethernet frame. Inner MAC addresses preserved. Used for intra-VLAN (same subnet) forwarding. One L2VNI per VLAN segment.
L3VNI
Carries IP packets — no inner L2 header. Used for inter-VLAN (different subnet) routed forwarding within a VRF. One L3VNI per VRF tenant.
Symmetric IRB — How It Works
Ingress VTEP: receives L2 frame on VLAN 10 access port → routes at L3 within VRF → strips L2 header → encapsulates IP payload with L3VNI in VXLAN header → sends to egress VTEP loopback.

Egress VTEP: receives on L3VNI → looks up destination IP in VRF routing table → finds host reachable on local VLAN 20 → adds new L2 header with destination MAC → delivers on VLAN 20 access port.

Both VTEPs do L3 routing — that is why it is called symmetric. The return path mirrors the forward path exactly, using the L3VNI in both directions.
Spine as BGP Route Reflector
Both spines run iBGP AS 65000 with route-reflector-client toward all leaf VTEPs. next-hop-unchanged preserves VTEP loopback as BGP next-hop. Spines reflect RT-2, RT-3, and RT-5 routes between all VTEP clients.
router bgp 65000 | route-reflector-client | next-hop-unchanged | send-community extended
